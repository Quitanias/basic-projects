# DevOps & SRE Local Simulation Environment 🚀

Welcome to my Reliability Engineering and Infrastructure Automation laboratory!

This project was built from scratch to simulate a complete and functional AWS environment, executed 100% locally. It demonstrates advanced skills in **Infrastructure as Code (IaC)**, **Configuration Management**, **Secrets Management**, and, most importantly, **Automated Testing (Continuous Verification)**.

The core idea is to showcase a real operational lifecycle (Day 0, Day 1, and Day 2) in a lean way, orchestrating industry-standard tools without the need to spend on public cloud during development.

---

## 🏗️ Architecture and Overview

The project is divided into clear domains of responsibility:

1. **Bootstrap (Environment Preparation)**: A _Bash_ script orchestrates the creation of Docker containers to simulate the ecosystem (LocalStack and HashiCorp Vault), serving as the foundation for the infrastructure.
2. **Provisioning (Terraform)**: Terraform modules provision simulated AWS resources (EC2, EBS, RDS) consuming dynamic credentials securely from Vault. The *state* is managed, and the code is structured around componentization.
3. **Configuration (Ansible)**: A suite of Ansible playbooks (organized in a _Roles_ and _Collections_ architecture) is triggered right after server creation. It installs and configures Nginx, Docker, FluentBit, Node Exporter, and applies local Firewall rules on Amazon Linux.
4. **Governance and Pipeline (CI/CD)**: A workflow configured in GitHub Actions performing dozens of continuous security validations and standard checks.

## 🧪 Testing Strategy (Shift-Left Testing)

The main complexity and differential of this repository lie in quality assurance. Unlike IaC projects that simply "apply" the code and blindly trust the execution, here we implement rigorous validations at both ends:

### 1. Testing Infrastructure: Terratest (Go) 🟩
Using the Gruntwork library, we wrote unit tests in **Golang** (`test/terraform_test.go`).
- **How it works:** The test initializes the Terraform module, runs `apply` passing the environment variables, and retrieves the *HCL Outputs*.
- **What we validate:** Strict assertions ensure that the EC2 instances spun up correctly with a valid Private IP, and that EBS volumes were indeed attached.
- Finally, a `defer Destroy` cleans up the environment, honoring TDD practices in infrastructure.

### 2. Testing Configuration: Molecule (Docker/Ansible) 🐳
The behavior of the Ansible *roles* is tested in isolation by **Molecule**.
- **How it works:** Molecule provisions a fresh `amazonlinux:2023` Docker image from scratch, applies the master playbook, and uses Ansible itself for validation.
- **What we validate:** The `verify.yml` file guarantees idempotency by checking for active services in *systemd* and ensuring zero exit shell codes for the app's crucial dependencies.

## 🚀 How to Run

Everything in the repository has been wrapped via `Make` to ensure the best development experience.

### 1. Requirements
- Docker and Docker Compose installed.
- Go (for Infrastructure tests).
- Python 3 / Pip (for Ansible and Molecule).
- Terraform CLI.
- Hashicorp Vault CLI.

### 2. Spinning up the infrastructure and testing

```bash
# 1. Spin up the base containers and generate the secrets
make bootstrap

# IMPORTANT: Load the Vault token into your current session
source scripts/bootstrap.sh

# 2. Initialize Terraform and generate resources
make init
make plan
make apply

# 3. Test Configuration automations (Ansible):
make test-ansible

# 4. Test Infrastructure automations (Terraform):
make test-tf

# 5. Destroy the infrastructure generated by Terraform
make destroy

# 6. Shut down simulator containers and perform a clean sweep
make clean
```
